# Sprint 9: MLS Rust NIF Contract TDD Plan

**Date**: 2026-02-25  
**Status**: Proposed  
**Depends On**: [9.0-mls-rust-nif-contract-deep-dive.md](9.0-mls-rust-nif-contract-deep-dive.md)

---

## Monorepo Layout Rule

1. Keep top-level `/native` reserved for future native app clients.
2. Implement MLS NIF under backend infrastructure:
   - `backend/infra/mls_nif/`
3. Keep Elixir-facing contract in:
   - `backend/lib/famichat/crypto/mls.ex`

---

## Decision

Yes, TDD-first is the right approach here.

Reason:

1. MLS + NIF work has high blast radius (crypto correctness + BEAM stability + state lifecycle).
2. The contract is now explicit, so we can encode it as failing tests before implementation details.
3. It enforces the single shared path principle (no special LLM/test-only runtime branch).

---

## Outcomes We Must Lock

1. Stable Elixir-facing contract for MLS operations.
2. Deterministic protocol invariants (pending proposals, commit/welcome ordering, rollback).
3. Fail-closed behavior on crypto/state errors (no plaintext fallback).
4. Observable SLO and telemetry gates before rollout.

---

## TDD Strategy (MECE)

### Track A: API and Error Contract Tests (Elixir Boundary)

Goal: lock wrapper API shape and error taxonomy before NIF implementation.

Red tests:

1. `backend/test/famichat/crypto/mls/adapter_contract_test.exs`
2. Assert each operation returns `{:ok, payload}` or `{:error, code, details}` only.
3. Assert unknown/invalid params map to `:invalid_input`.
4. Assert any crypto failure maps to `:crypto_failure`.
5. Assert no fallback path returns plaintext.

Green implementation:

1. Add `Famichat.Crypto.MLS` behavior and typed wrapper module.
2. Add stub adapter returning contract-compliant failures until NIF lands.

### Track B: Protocol Invariant Tests (State Machine)

Goal: lock MLS lifecycle semantics independent of transport/UI.

Red tests:

1. `backend/test/famichat/crypto/mls/protocol_invariants_test.exs`
2. `pending_proposals -> create_application_message` returns `{:error, :pending_proposals, ...}`
3. `staged_commit` must be validated before `merge_staged_commit`.
4. Welcome processed before commit acceptance is rejected.
5. Rejected commit requires `clear_pending_commit` and leaves state usable.

Green implementation:

1. Implement wrapper guardrails + state transition enforcement.
2. Add explicit helper functions for inspect/approve/merge/rollback.

### Track C: Storage and Recovery Semantics Tests

Goal: lock persistence invariants and forward-secrecy deletion rules.

Red tests:

1. `backend/test/famichat/crypto/mls/storage_contract_test.exs`
2. Missing required group state returns `:storage_inconsistent`.
3. Deleted key material cannot be loaded again.
4. Key package consumption/replenishment thresholds are enforced.
5. State-loss recovery path (rejoin) is deterministic and audited.

Green implementation:

1. Add storage adapter interfaces (in-memory test adapter first).
2. Add persistent adapter with explicit delete semantics.

### Track D: Messaging Integration Contract Tests

Goal: ensure existing message/channel path remains canonical and fail-closed.

Red tests:

1. `backend/test/famichat/chat/message_service_mls_contract_test.exs`
2. `send_message/1` encrypt-path failure returns error and persists nothing.
3. `get_conversation_messages/2` decrypt-path failure is explicit and redacted.
4. Non-200 flows produce no broadcast side effects.
5. Legacy plaintext paths are not silently used when MLS is required.

Green implementation:

1. Integrate `Famichat.Chat.MessageService` with MLS wrapper.
2. Keep telemetry and redaction contract stable.

### Track E: Performance and Rollout Gate Tests

Goal: prevent shipping without required visibility and baseline performance checks.

Red tests:

1. `backend/test/famichat/crypto/mls/telemetry_contract_test.exs`
2. Assert emission for:
   - `send_application_message`
   - `process_application_message`
   - `mls_commit/update/add/remove`
3. Assert encryption failures emit non-sensitive metadata only.
4. Assert required metrics are present before feature flag can be enabled beyond dev/test.

Green implementation:

1. Add telemetry spans and metric keys in wrapper/orchestration layer.
2. Gate rollout on telemetry contract pass.

---

## Sequencing

1. Build Tracks A+B first (contract and invariants).
2. Add Track C (storage semantics).
3. Wire Track D (message-service integration).
4. Add Track E (operational gates) before non-dev rollout.

This ordering minimizes rework and prevents transport/UI coupling before core correctness.

## Done Criteria (TDD Phase Complete)

1. All Track A-E contract tests are green.
2. No plaintext fallback path exists when encryption is required.
3. Rollback and pending-proposal invariants are enforced.
4. Required telemetry fields exist and are redaction-safe.
5. Existing channel/API path remains the single production path.

---

## Non-Goals for This Phase

1. Full UX polish for MLS flows.
2. Federation or cross-neighborhood protocol features.
3. Optional alternate protocol support.
