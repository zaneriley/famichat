## Why this plan exists

Story 7.4.2 asks for a CLI-testable endpoint that can trigger channel broadcasts and be verified with `curl`.
The codebase currently has overlapping test-broadcast endpoints, inconsistent route gating, and incomplete test/docs alignment.
This plan defines a single safe, reproducible workflow that satisfies both product intent and security expectations.

## Goals

1. Provide one canonical CLI broadcast endpoint with deterministic behavior.
2. Require auth and enforce conversation-level authorization (no arbitrary topic fanout).
3. Keep quick developer verification via `curl` practical.
4. Preserve compatibility briefly while converging on one endpoint.
5. Align sprint docs so status is internally consistent.

## Non-goals

1. Building full production event-publishing APIs.
2. Replacing Phoenix Channels with HTTP polling or SSE.
3. Solving full auth UX for operators beyond a minimal dev/test token path.

## Spirit check (not just checkbox compliance)

This plan prioritizes a real operational drill:

1. A developer can authenticate, run one `curl`, and observe delivery from a subscriber.
2. The endpoint cannot be used as an unauthenticated or arbitrary broadcast backdoor.
3. Success and failure both produce testable, outcome-focused evidence.

## Proposed design

### 1) Canonical endpoint + compatibility window

1. Canonical route: `POST /api/test/broadcast`.
2. Keep `POST /api/test/test_events` as a temporary alias for one sprint.
3. Alias response includes deprecation signaling (for example: deprecation/sunset headers).
4. Removal target for alias: Sprint 8 cleanup.

### 2) Route gating and auth

1. Move/keep test broadcast routes under explicit dev/test routing guard.
2. Use `:api_authenticated` pipeline so Bearer auth is mandatory.
3. Resolve router config inconsistency by using one app key for dev-routes checks.

### 3) Request and response contract

1. Request body:
   - `conversation_type` (`self|direct|group|family`)
   - `conversation_id` (UUID)
   - `body` (string)
   - Optional encryption metadata: `encryption_flag`, `version_tag`, `key_id`
2. Server derives topic from type/id; clients never submit raw topic.
3. Server emits `"new_msg"` event payload matching channel expectations.
4. Response contract:
   - `200`: accepted + emitted metadata
   - `401`: missing/invalid bearer token
   - `403`: authenticated but not authorized for conversation
   - `422`: validation error

### 4) Authorization behavior

1. Caller must be a member of target conversation.
2. Reuse existing membership/authorization paths where possible.
3. Failed authorization must not broadcast side effects.

### 5) Developer ergonomics

1. Add a documented dev/test token acquisition path (script or explicit command recipe).
2. Provide a copy-paste `curl` example and expected response.
3. Provide a subscriber verification recipe (channel test client or logs).

## Implementation plan

### Phase 1: Route and controller consolidation

1. Consolidate behavior into one controller path.
2. Keep alias route for compatibility and emit deprecation hints.
3. Ensure route is available only in dev/test.

### Phase 2: Validation + authorization hardening

1. Validate payload shape and UUID/type rules.
2. Authorize caller against conversation membership.
3. Broadcast only on fully valid/authorized requests.

### Phase 3: Test coverage

Add/refresh controller tests to cover:

1. `200` success emits one broadcast with expected payload.
2. `401` without bearer token.
3. `403` when caller not participant.
4. `422` invalid payload/metadata.
5. No-broadcast assertions for all non-200 paths.
6. Alias route parity + deprecation header assertions (during compatibility window).

## Delegated execution tracks

### Agent A: Router and auth pipeline hardening

1. Scope test-broadcast routes under explicit dev/test guard.
2. Enforce bearer auth pipeline for canonical + alias routes.
3. Keep compatibility alias route active for one sprint.
4. Deliverables:
   - Router diff
   - Proof that missing/invalid token returns `401`

### Agent B: Controller contract and authorization behavior

1. Canonicalize request fields to `conversation_type`, `conversation_id`, `body`.
2. Derive topic server-side only; reject arbitrary topic inputs.
3. Enforce conversation membership authorization (`403` on non-member).
4. Emit deprecation headers on alias route responses.
5. Deliverables:
   - Controller diff
   - Response examples for `200/403/422`

### Agent C: Contract tests and verification gates

1. Add integration/controller tests for `200/401/403/422`.
2. Assert exact side effects:
   - success -> exactly one broadcast
   - all non-success -> zero broadcasts
3. Add alias parity tests and deprecation header assertions.
4. Run `elixir:test`, `elixir:lint`, `elixir:security-check`, `elixir:static-analysis`.
5. Deliverables:
   - Test diff
   - Command output log summary

## Test matrix

1. `authorized valid payload -> 200 + broadcast once`
2. `missing token -> 401 + no broadcast`
3. `invalid token -> 401 + no broadcast`
4. `valid token, unauthorized conversation -> 403 + no broadcast`
5. `invalid conversation_type/id/body -> 422 + no broadcast`
6. `alias endpoint -> same semantics + deprecation headers`

## Risks and mitigations

1. **Risk**: Breaking existing internal scripts on `/api/test/test_events`.  
   **Mitigation**: Keep alias for one sprint, emit deprecation headers, document sunset date.

2. **Risk**: Auth requirement hurts quick local testing.  
   **Mitigation**: Provide minimal token acquisition helper in docs.

3. **Risk**: Topic derivation mismatch with channel expectations.  
   **Mitigation**: Reuse the same type/id composition used in channel tests.

## Definition of done

1. Single canonical CLI broadcast endpoint is implemented and tested.
2. Endpoint is auth-protected and membership-authorized.
3. Non-success paths prove no broadcast side effects.
4. Compatibility alias exists with explicit deprecation behavior.
5. `CURRENT-SPRINT`, `STATUS`, and `ROADMAP` are mutually consistent on 7.4.2 state.
6. A developer can follow one documented `curl` workflow and verify end-to-end delivery.
