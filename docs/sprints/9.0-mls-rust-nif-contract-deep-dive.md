## Executive Verdict

We have enough to begin implementation, but only after locking the contract below.

Main conclusion:

1. Direction is correct (MLS-first + OpenMLS).
2. The current plan is missing several protocol and boundary invariants that must be explicit before coding.
3. Those gaps are now documented here as a MECE implementation contract.

## Review Method (5 Parallel Tracks)

This review was run as five independent tracks and then reconciled:

1. **Agent A**: BEAM/NIF safety and Rustler boundary discipline
2. **Agent B**: MLS protocol correctness and message lifecycle
3. **Agent C**: Storage/key lifecycle and state recovery
4. **Agent D**: Delivery/authz model and product-policy integration
5. **Agent E**: Performance, dependency hygiene, and operational safety

---

## Agent A: NIF Boundary Findings

What is correct:

1. Rustler is a viable bridge for safe Rust NIF integration.
2. Current architecture expects Elixir as domain orchestrator and Rust NIF as crypto backend.

Gaps to fix:

1. **Scheduler policy is not explicit**.
2. **NIF side effects are not constrained** (must be no DB/network/file IO in the NIF path).
3. **Error surface is not standardized** (must be stable, typed, and redaction-safe).

Contract implications:

1. Any NIF call that can exceed ~1ms must be scheduled as dirty (`DirtyCpu` for crypto-heavy work).
2. Dirty I/O schedulers should only be used for actual blocking IO. Prefer no IO in NIF.
3. Use Rustler resources for long-lived state handles; keep mutable shared state behind synchronization primitives.
4. Never rely on panics for control flow; always return explicit error tuples.

---

## Agent B: MLS Protocol Findings

What is correct:

1. MLS/OpenMLS is aligned with group-scale roadmap.
2. Current docs already track commit/update/add/remove as separate performance class.

Gaps to fix:

1. **Pending proposal gating is not encoded in Famichat contract docs**.
2. **Commit inspection/merge flow is not fully specified**.
3. **Wire-format policy is not locked** (public vs private handshake messages).
4. **Welcome delivery rule is not locked** (must follow accepted Commit).

Contract implications:

1. If pending proposals exist, app-message creation must fail until a commit is made.
2. Every staged commit must be inspected and policy-validated before merge.
3. Welcome messages must not be forwarded/applied before associated commit acceptance.
4. Default to private wire-format policy unless explicitly overridden with documented risk acceptance.

---

## Agent C: Storage and Key Lifecycle Findings

What is correct:

1. You already track key/group lifecycle as Sprint 9 scope.
2. Cloak-backed persistence strategy is directionally aligned.

Gaps to fix:

1. **StorageProvider semantics are not translated into app-level invariants**.
2. **Deletion guarantees for forward secrecy are not called out as hard requirements**.
3. **Commit rejection rollback contract is not explicit**.
4. **State-loss recovery path exists conceptually but not yet in implementation plan detail**.

Contract implications:

1. Group-id keyed storage values required by OpenMLS must never resolve to missing on load paths.
2. Deleted MLS key material must be irrecoverably removed (no soft-retained copies).
3. On rejected/abandoned commit, clear pending commit and related temporary artifacts.
4. Recovery-from-state-loss must be a first-class flow (rejoin + membership reconciliation).

---

## Agent D: Delivery and Authorization Findings

What is correct:

1. Existing `7.4.2` secured CLI broadcast flow proves good authz discipline.
2. DS/AS roles are directionally consistent with MLS architecture.

Gaps to fix:

1. **ACL policy consistency across clients is not yet explicit in MLS implementation plan**.
2. **Proposal/Commit ordering expectations are not tied to DS behavior contract**.
3. **Multi-device model (client identity vs user identity) is not locked**.

Contract implications:

1. Authorization policy decisions for group changes must be deterministic and consistent across clients.
2. DS contract must preserve proposal visibility before dependent commits where required.
3. Multiple devices must be treated as multiple MLS clients, with explicit product semantics.

---

## Agent E: Performance and Security Ops Findings

What is correct:

1. Performance doc now separates steady-state app messages from group lifecycle costs.
2. You already track churn/tree-health sensitivity.

Gaps to fix:

1. **Dependency policy is not yet concrete enough for OpenMLS advisories**.
2. **Version floor is not pinned in docs**.
3. **Operational SLO gates are listed but not yet tied to rollout checkpoints**.

Contract implications:

1. Set OpenMLS minimum version policy to patched ranges (not vulnerable ranges).
2. Block rollout if required MLS telemetry is missing.
3. Track and alert on commit/update/remove latency and epoch drift before broader dogfooding.

## Unified MECE Contract (Implementation Lock)

### 1) Boundary Contract (Elixir vs Rust NIF)

Elixir owns:

1. Authentication/authorization and policy decisions
2. Request validation and API contracts
3. DB orchestration and business telemetry
4. Fanout orchestration and retry/circuit behavior

Rust NIF owns:

1. MLS cryptographic state transitions
2. Message encrypt/decrypt and protocol validation
3. Deterministic serialization/deserialization of MLS artifacts

Hard rule:

1. No network/DB IO inside NIF code path.
2. No alternate/LLM-only runtime path.
3. Fail closed on cryptographic errors (no plaintext fallback).

### 2) NIF API Contract (Minimal v1)

All calls return `{:ok, payload}` or `{:error, code, details}`.

Minimum operations:

1. `nif_version/0` and `nif_health/0`
2. `create_key_package/1`
3. `create_group/1`
4. `join_from_welcome/1`
5. `process_incoming/1`
6. `commit_to_pending/1`
7. `merge_staged_commit/1`
8. `clear_pending_commit/1`
9. `create_application_message/1`
10. `export_group_info/1` and `export_ratchet_tree/1`

Error taxonomy buckets:

1. `:invalid_input`
2. `:unauthorized_operation`
3. `:stale_epoch`
4. `:pending_proposals`
5. `:commit_rejected`
6. `:storage_inconsistent`
7. `:crypto_failure`
8. `:unsupported_capability`

### 3) Protocol Invariants Contract

1. Pending proposals must be committed before creating app messages.
2. Staged commits must be inspected and policy-approved before merge.
3. Commit proposal lists must be validated for logical and policy consistency.
4. Welcome processing must only occur after associated commit acceptance.
5. Credential validation occurs on join and during message processing (add/update paths).

### 4) Storage Contract

1. OpenMLS-required group state fields must be load-consistent.
2. Delete operations for key material must be irreversible.
3. KeyPackage lifecycle must include consumption/replenishment policy.
4. Rejected commit handling must restore pre-stage state (`clear_pending_commit`) and clean temporary material where required.

### 5) Delivery and Access Control Contract

1. DS role is delivery + key package directory; DS is not trusted for group-key confidentiality.
2. Client policy must enforce consistent access-control decisions.
3. Proposal/commit ordering assumptions must be codified in transport handling.
4. Multi-device behavior must treat each device as a distinct MLS client identity under a user umbrella.

### 6) Wire-Format and Feature Policy Contract

Default:

1. Private wire format for handshake and application messages.
2. Standalone proposals disabled unless explicitly required.

If public messages/proposals are enabled:

1. Must remain on patched OpenMLS versions.
2. Must have explicit additional validation tests in CI.

### 7) Performance and Scheduler Contract

1. Crypto-heavy NIFs use dirty CPU scheduling.
2. NIF calls must be classified CPU vs IO correctly.
3. Steady-state app-message SLO and group-lifecycle SLOs are tracked separately.
4. Rollout gates require p50/p95/p99 visibility for:
   - `send_application_message`
   - `process_application_message`
   - `mls_commit/update/add/remove`
   - epoch drift and group state growth

### 8) Dependency and Security Contract

1. Do not use OpenMLS vulnerable ranges:
   - `<= 0.7.0` (GHSA-qr9h-x63w-vqfm)
   - `< 0.7.2` (GHSA-8x3w-qj7j-gqhf)
2. Pin to a patched supported baseline and review monthly.
3. High/critical advisories trigger patch SLA and release gating.

### 9) Test Contract (Outcome-Focused)

Must-have characterization tests:

1. Pending proposal prevents app-message send until commit.
2. Rejected commit rollback leaves group state usable and consistent.
3. Welcome-before-commit acceptance is rejected.
4. Credential validation failures are surfaced deterministically.
5. Unauthorized membership changes are rejected with no side effects.
6. Out-of-order delivery tolerance behaves per configured ratchet limits.
7. Replay/truncation negative cases are rejected in patched dependency baselines.
8. No plaintext fallback on encryption/decryption failure.
